# Internationalization (i18n)

## Supported Languages

- **Hungarian (hu)** — default
- **English (en)**

## Routing

Locale-prefix routing:
- `/hu/kavek` — Hungarian
- `/en/coffees` — English
- `/` → redirect `/hu`

URL structure (HU → EN):
```
/kavek              → /coffees
/kavek/[slug]       → /coffees/[slug]
/eszkozok           → /equipment
/merch              → /merch
/csomagok           → /bundles
/sztorik            → /stories
/sztorik/[slug]     → /stories/[slug]
/kosar              → /cart
/penztar/...        → /checkout/...
/elofizetes         → /subscription
/belepes            → /login
/regisztracio       → /register
/fiokom/...         → /account/...
```

Admin routes: `/admin/...` (no locale prefix, always Hungarian).

## Language Switcher

In the header, globally accessible:
- Button: "HU" / "EN" toggle (or dropdown)
- On click: navigates to the same page in the other locale
- Language selection persists: in the user profile (logged in), in a cookie (anonymous)

## Content i18n

All content (product names, descriptions, stories) is stored in both languages. The appropriate language version is displayed on the UI based on the current locale.

All user interface text (buttons, labels, error messages) must also be available in both languages.

## Email i18n

All emails are sent according to the user's language preference:
- Logged-in user: profile language
- Anonymous (e.g., gift card recipient): default HU
- Both email subject and body follow the language setting

## Admin Language

The admin interface is Hungarian only. There is no language switcher in the admin section. However, in content editor forms, both language fields are editable (HU tab / EN tab).

## Translation File Layout (sidecar pattern — IMPORTANT for parallel changes)

Translation messages MUST be split into per-feature sidecar files under `messages/<locale>/<feature>.json`, NOT a single monolithic `messages/hu.json` / `messages/en.json`. This avoids merge conflicts when multiple changes add keys in parallel.

Layout:
```
messages/
  hu/
    common.json        # shared (header, footer, nav, generic buttons)
    catalog.json       # product list/detail/search/filter
    cart.json          # cart + checkout
    auth.json          # registration + login + password reset
    account.json       # profile, addresses, orders, notifications
    subscription.json
    admin.json         # admin panel (HU only — see "Admin Language")
    promotions.json    # coupons + promo days + gift cards
    email.json         # email templates
    stories.json       # content/blog
    reviews.json       # reviews + wishlist
    seo.json           # meta titles/descriptions
  en/
    <same structure, no admin.json>
```

Loader: `next-intl` is configured to merge all sidecar files into a single namespace tree at request time. Keys are namespaced by file (e.g., `cart.checkout.payButton`).

**Rules for agents:**
- Each change adds keys ONLY to its own feature sidecar (e.g., the auth change touches only `messages/{hu,en}/auth.json`).
- NEVER delete or rewrite another feature's sidecar — even if it looks unused. The other feature owns it.
- NEVER consolidate sidecars into a single file. The split is intentional for merge-conflict avoidance.
- If a key is genuinely shared across features (e.g., "Save" button), put it in `common.json`.
- Adding a brand-new sidecar (new feature) is fine; mirror it in both `hu/` and `en/`.
