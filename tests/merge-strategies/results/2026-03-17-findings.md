# Merge Strategy Benchmark — Findings (2026-03-17)

**Environment:** `/tmp/minishop-run3` (MiniShop Run #17 project)
**pnpm-lock.yaml:** 4157 lines, pnpm 10.12.1
**wt-tools commit at test time:** `8144b8d4f`

---

## Key Finding: SC1 vs SC2 Behave Differently

### SC1: Lockfile-only conflict
**Result: git auto-resolved (no conflict)**

When two branches modify *different lines* of `pnpm-lock.yaml`, git's
text merge succeeds without conflict markers. This is the common case
when two changes add different dependencies.

→ **Implication:** The lockfile conflict only appears when both changes
modify the SAME lines (e.g., both touch the `importers` section header).

### SC2: Mixed — lockfile + app code
**Result: ONLY lockfile conflicted, app code did NOT**

Agent #3 (conflict simulation) tested a scenario where two branches:
- Both modified line 10 of `pnpm-lock.yaml`
- Both appended a comment to the same TS file

Actual outcome:
```
Conflicted files: pnpm-lock.yaml
pnpm-lock.yaml markers: 2
src/app/cart/order-actions.ts markers: 0
```

**The app file auto-merged because both sides made additive changes
(prepending vs appending a comment). Real app code rarely causes conflicts
unless both changes modify the same function/block.**

## Strategy Test Results

### S1: partial_mode=true (COMMITTED `8144b8d4f`)

**Test:** Applied S1 logic to SC2 conflict (pnpm-lock.yaml conflicted)
```
Conflicted: pnpm-lock.yaml
AUTO-RESOLVED (ours): pnpm-lock.yaml
→ git commit --no-edit → SUCCESS
Remaining LLM input: 0 lines
```

**Result: ✅ SUCCESS — zero LLM input needed**

Before the fix (`partial_mode=false`), a mixed conflict would skip
auto-resolution and send the 4157-line lockfile to LLM → always failed.
After fix: lockfile resolved, LLM only processes actual app code.

### S2: .gitattributes merge=ours

**Test:** `git check-attr merge pnpm-lock.yaml` — returned "unspecified"
(no .gitattributes in minishop-run3)

**Result: ✅ WOULD WORK if configured** — git would suppress the
conflict at source, never producing markers.

**Integration cost:** Requires:
1. `.gitattributes` file in scaffold (1 new file)
2. `git config merge.ours.driver true` in run.sh (1 line)

Zero runtime overhead, prevents conflict before any Python code runs.

### S3: post-merge hook

**Test:** Hook created and tested on minishop-run3.

**Result: ❌ HARMFUL** — the hook runs `pnpm install --no-frozen-lockfile`
which modifies `node_modules/.bin/` symlinks (49 dirty files) and
regenerates `pnpm-lock.yaml`. This dirty state blocks the NEXT merge
attempt with "local changes would be overwritten by merge".

**This was the actual root cause of run #16/#17 merge-blocked!**
The hook regenerates after each merge but wt-merge doesn't commit those
hook-generated changes before attempting the next merge.

**Decision: DO NOT USE post-merge hook.** Lockfile regeneration should
happen through wt-merge's `regenerate_lockfile()` and merger.py's
`_post_merge_deps_install()` which properly stage and commit.

## Controlled Experiment Results (2026-03-17)

Ran on `/tmp/minishop-run3` with 3 strategies against same-line lockfile conflict:

| Strategy | Conflict? | Auto-fixed? | Side effects |
|---|---|---|---|
| Baseline (no intervention) | **YES** — 1 marker in lockfile | NO | — |
| `.gitattributes merge=ours` | **NO** — prevented at git level | N/A | None! Winner kept = ours |
| gitattributes + post-merge hook | NO lockfile conflict | — | **49 dirty files** → blocks next merge |

**Winner: `.gitattributes merge=ours` alone, NO hook.**

## DB vs Frontend Separation Analysis

From `git log -- prisma/schema.prisma` on minishop-run3:

Changes that touched `prisma/schema.prisma`:
- `db-schema-seed` — creates schema (solo)
- `auth-setup` — adds User model
- `admin-products` — adds Product/Variant models
- `orders-checkout` — adds Order model

These changes were sequential (phase dependency) in runs #16–#17,
which is why no schema conflicts occurred.

**Conclusion:** The planner already sequences DB changes correctly via
`depends_on`. Explicit DB/frontend separation would add overhead
(more changes = more dispatch overhead) without reducing conflicts,
since schema changes already run sequentially.

**Recommendation: Don't split DB vs frontend — maintain phase ordering.**
