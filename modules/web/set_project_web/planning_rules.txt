Change sizing and decomposition granularity:
- Each change SHOULD target 50-150K output tokens. With 1M context windows, changes up to 200K are acceptable but larger changes have more merge conflict risk.
- NEVER combine infrastructure/foundation with authentication in one change. Foundation sets up: Prisma schema, seed data, Tailwind, layout, tooling. Auth sets up: NextAuth, middleware, login/register pages. These are separate concerns with separate test surfaces.
- NEVER combine cart and checkout in one change. Cart (session, add/remove/update) and checkout (order placement, payment, order history) are separate features.
- Admin CRUD can be one change if it covers a single entity (products). If it covers multiple entities (products + orders + users), split by entity.
- Prefer 3 phases over 4+. More phases = more sequential bottlenecks. Ideal: Phase 1 (foundation + auth parallel), Phase 2 (catalog + cart + admin parallel), Phase 3 (checkout + polish).
- Each change MUST have its own E2E test file. Test file names SHOULD match the change name (e.g., change `shopping-cart` → `tests/e2e/cart.spec.ts`).

Template file protection:
- Files deployed by `set-project init` (globals.css, vitest.config.ts, playwright.config.ts, tsconfig.json, next.config.js, postcss.config.mjs) SHOULD NOT be modified unless the change specifically requires it. If a config change is needed, document WHY in the commit message.
- `src/lib/prisma.ts` and `src/lib/utils.ts` are template files — use them as-is, do not recreate or rename.

Playwright E2E test planning (when e2e_command is configured):
- The infrastructure/foundation change (first in dependency order) MUST set up Playwright alongside the test runner (Jest or Vitest):
  * Create playwright.config.ts with PW_PORT env var support, webServer auto-start, headless: true, and screenshot: 'on' in the use section (headless: true prevents browser windows opening on desktop machines; screenshot: 'on' ensures PNG capture instead of text-only error-context.md)
  * Set `reuseExistingServer: false` in playwright.config.ts webServer — stale dev servers from previous gate runs cause flaky tests
  * Set `workers: 2` (not unlimited) — unlimited workers overload Next.js dev server compilation, causing timeout failures
  * Set Playwright `locale` matching the project's i18n default (e.g., `locale: 'hu-HU'` for Hungarian projects) — locale mismatch causes wrong redirects in i18n middleware
  * If using Vitest: add `exclude: ["**/node_modules/**", "**/tests/e2e/**"]` to vitest.config.ts AND use `"test": "vitest run"` (NOT `"test": "vitest"` — watch mode hangs in CI/gates)
  * If using Jest: add testPathIgnorePatterns: ["/node_modules/", "/tests/e2e/"] to jest.config (Jest crashes on Playwright .spec.ts imports in jsdom)
  * Add @playwright/test to devDependencies + run npx playwright install chromium
  * Create tests/e2e/global-setup.ts: prisma generate → prisma db push --force-reset → prisma db seed
- Each subsequent feature change creating a user-facing route MUST create tests/e2e/<feature>.spec.ts as an explicit file deliverable in scope (e.g., "Create tests/e2e/cart.spec.ts")
- Include CONCRETE test scenarios per feature:
  * Page URL to visit
  * User interactions: what to click, what data to type, what to select
  * Expected outcomes: visible text, redirects, error messages
  * Auth scenarios: verify protected routes redirect unauthenticated users
  * Error scenarios: invalid form input, missing required fields
  * COLD-VISIT test: navigate directly to the page as the FIRST action (no prior login, no add-to-cart, no session cookie). This catches Server Component cookie/session bugs where cookies().set() crashes outside a Server Action.
- Example scope:
  Create tests/e2e/orders.spec.ts:
  - Cold visit: go to /cart directly (no prior actions) → should show empty state, NOT crash
  - Visit /products → click "Add to Cart" on first product → cart badge shows "1"
  - Visit /cart → verify product name and price → click "Checkout" → fill form → verify redirect to /orders/[id]
  - Visit /admin/orders without login → verify redirect to /admin/login
- Do NOT defer all E2E tests to a consolidation change — this overloads a single agent. Each feature change owns its tests.
- Do NOT just list "Functional test scenarios:" as descriptions — create actual test files.
- E2E tests run pre-merge in the worktree via e2e_command against an auto-started dev server with isolated port and DB.
- E2E test stability (REQUIRED patterns to prevent flaky tests):
  * ALWAYS use `waitUntil: 'networkidle'` on `page.goto()` before interacting with forms — React hydration races cause native form GET submissions instead of Server Action POST calls
  * Use `page.waitForLoadState('networkidle')` after navigation to pages with client components
  * Set generous timeouts for `waitForURL` assertions (at least 10s) — Next.js dev server compiles routes on first visit, which can take 3-5s
  * Test selectors MUST match actual component data-testid values — verify by reading the component source, not guessing
- Build script MUST handle missing database:
  * The `build` script in package.json MUST include `prisma generate && prisma db push --accept-data-loss &&` before `next build` — the build gate runs in fresh worktrees where no database tables exist yet
  * Any page that queries the database at build time (sitemap.ts, static generation) MUST wrap DB queries in try/catch to handle missing tables gracefully
  * Without this, the build gate enters an infinite retry loop: build fails → agent retries → build fails again (no code change fixes it because the schema just isn't pushed)
- CRUD E2E test completeness (REQUIRED for feature changes with data mutations):
  * CREATE: fill form with valid data → submit → verify new item appears in list
  * READ: navigate to list → verify items render with correct data
  * UPDATE: click edit → change a field → submit → verify change reflected
  * DELETE: click delete → confirm dialog → verify item removed from list
  * VALIDATION: submit with invalid/empty data → verify error messages shown
  * LAYOUT: verify sidebar/nav present on every page in the route group (catches layout bugs where child pages override the shared layout)
  Tests that only verify "page loads without error" are INSUFFICIENT — they miss runtime bugs in Server Actions, form submissions, and data mutations.

Parallel agent coordination (multi-worktree builds):
- When a foundational change creates a shared layout structure (route groups, layout components, providers), the planner MUST specify the exact path in downstream change scopes. Example: if auth-navigation creates `src/app/(shop)/layout.tsx` with a navigation header, product-catalog's scope must say "Create pages under `src/app/(shop)/products/`" — NOT just "Create product listing page."
- Route group decisions belong to the foundational/navigation change. Downstream changes MUST reference the established route group by path, not invent their own placement.
- Shared components (NavigationHeader, Footer, Sidebar) are created by one change — downstream changes use them, never recreate them.
- Database schema changes belong to ONE change (typically the foundational). Other changes that need new models MUST list them as dependencies and specify the exact model names in scope so the planner sequences correctly.
- When two changes both need to modify the same file (e.g., prisma/schema.prisma, middleware.ts), the planner MUST sequence them with explicit dependency, never run them in parallel.

Design fidelity — concrete UI elements in scope (when design-snapshot.md exists):
- When a design snapshot or Figma source is available, the planner MUST extract concrete UI elements from the design and list them explicitly in each change's scope. Do NOT write abstract scope like "Create product listing page" — instead write "Product card: product image, name, price range (formatted as '$X.XX – $Y.YY'), 'View Details' button linking to /products/[id]."
- Navigation elements, back-links, and breadcrumbs visible in the design MUST be listed as explicit deliverables. Example: "Product detail page includes 'Back to Products' link at top, linking to /products."
- Button labels, empty states, error messages, and badge text from the design MUST appear verbatim in the scope. Agents cannot implement what they don't know exists.
- If the design shows a component (e.g., cart badge with item count in the header), specify which change owns it and which change consumes it.

Placeholder images and static assets:
- When seed data or components reference product/content images, use placeholder URLs — NEVER reference local files like `/images/product.jpg` that don't exist in the repo.
- Preferred approaches (in order):
  1. Placeholder service: `https://placehold.co/400x300/EEE/999?text=Product+Name` — works immediately, customizable text/color
  2. Inline SVG data URI: `data:image/svg+xml,...` — no external dependency, works offline
  3. Next.js Image with `unoptimized` prop + placeholder URL — for when using next/image component
- The foundation/seed change MUST ensure all imageUrl fields in seed data use working URLs, not paths to non-existent local files.
- If the project has a `public/images/` directory with actual files, reference those. Otherwise use placeholder URLs.

Test runner setup (vitest / jest):
- The infrastructure/foundation change MUST ensure package.json includes a `test` script (e.g., `"test": "vitest run"`) and vitest (or jest) in devDependencies. Do NOT defer test runner setup to later changes — the foundation change owns all tooling.
- If the project uses vitest, the foundation change MUST install vitest and create a vitest.config.ts (or vite.config.ts with test config). This is a blocking requirement — integration gates will fail if the test script is missing.

Application Startup guide maintenance:
- Infrastructure/foundational changes that add new setup steps (DB, E2E tooling, env vars, build commands) MUST update the `## Application Startup` section in CLAUDE.md. This section is the canonical reference for agents entering the worktree mid-project. If you add a Prisma DB, document `npx prisma db push`. If you add Playwright, document `npx playwright install --with-deps chromium`. Keep it current.

Framework-specific security patterns (Next.js / React):
- Route protection: create middleware.ts at project root that checks auth for protected route prefixes (/admin, /dashboard, /account). Do NOT put auth checks inside individual route handlers.
- Prisma ownership patterns: every findUnique/findFirst/update/delete on user-scoped data MUST include the owning entity in the where clause (e.g., { id: params.id, userId: session.user.id }), never just { id: params.id }.
- Server Actions: validate all inputs with Zod schemas before database operations. Never pass raw formData fields directly to Prisma.
- CSRF: Next.js Server Actions have built-in CSRF protection — do NOT disable it. For API routes using cookies, verify SameSite=Lax is set.
- XSS: never use dangerouslySetInnerHTML with user-supplied content. If rich text is needed, sanitize with DOMPurify on the server before rendering.
