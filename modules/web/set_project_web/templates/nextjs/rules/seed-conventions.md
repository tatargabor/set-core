# Seed Conventions

`prisma/seed.ts` runs on every E2E bootstrap (`prisma db push --force-reset && prisma db seed`). It must be deterministic and idempotent so that a re-seed from a non-empty database produces the same rows as a seed from an empty one. Non-idempotent seeds crash or silently duplicate rows whenever two changes touch the same seed file and the test harness re-runs it.

## Idempotency Rule — Guard Every `create`

Every `prisma.X.create()` (or `createMany`) MUST be preceded by a `findFirst`/`findUnique` guard on a deterministic key. `upsert` is the canonical shape — use it.

**Wrong — blind `create` duplicates on re-seed:**
```typescript
// Re-run creates a second "admin" user, then unique email constraint crashes
await prisma.user.create({
  data: { email: "admin@example.com", name: "Admin", role: "ADMIN" },
});

// Re-run stacks duplicate orders on the same user
await prisma.order.create({
  data: { userId: user.id, status: "DELIVERED", total: 2500 },
});
```

**Correct — `upsert` for records with a natural unique key:**
```typescript
await prisma.user.upsert({
  where: { email: "admin@example.com" },
  update: {},
  create: { email: "admin@example.com", name: "Admin", role: "ADMIN" },
});
```

**Correct — `findFirst` guard for records without a unique key:**
```typescript
const existing = await prisma.order.findFirst({
  where: { userId: user.id, status: "DELIVERED", totalCents: 250000 },
});
if (!existing) {
  await prisma.order.create({
    data: { userId: user.id, status: "DELIVERED", totalCents: 250000 },
  });
}
```

**The rule:** every mutating operation in `seed.ts` MUST be either `upsert` (preferred) or `findFirst` + `create`. Pure `create` is allowed only for tables that were just truncated in the same seed run AND that the rest of the seed does not depend on by natural key.

## Deterministic Keys

The lookup key for a guard must be deterministic across runs — do NOT use `Date.now()`, `Math.random()`, UUIDs, or auto-generated IDs as the uniqueness basis.

- Users → email
- Products → slug
- Categories → name or slug
- Orders (for seed fixtures) → a composite of `(userId, status, createdOffsetDays)` or a dedicated `seedKey` column

If a record genuinely has no natural key, add a `seedKey String @unique` column used only by seed fixtures.

## Slugs Must Be Deterministic and Match Tests

Seed slugs MUST be generated by a pure function of the source field (e.g., `slugify(name)`) with no locale-dependent transliteration. E2E tests source slugs from the seed (see testing-conventions.md § "Test Data MUST Come From Seed") — if the seed generates `ethiopia-yirgacheffe` but tests hardcode `etiopia-yirgacheffe`, every `/products/[slug]` test 404s.

- Pick ONE canonical language for seed slugs (usually English) and document it at the top of `seed.ts`.
- Use a stable slug library (`slugify`, `@sindresorhus/slugify`) — do not hand-roll locale transliteration.
- Never compute a slug from a translated/localised label; always from the base field.

## Required Test User

Every seed MUST create a test user with known credentials, documented at the top of the file:

```typescript
// TEST USER — used by E2E login tests.
// Email: test@example.com
// Password: password123
const TEST_USER_EMAIL = "test@example.com";
const TEST_USER_PASSWORD = "password123";

const passwordHash = await bcrypt.hash(TEST_USER_PASSWORD, 10);
await prisma.user.upsert({
  where: { email: TEST_USER_EMAIL },
  update: { passwordHash },  // update on re-seed in case the hash algorithm changed
  create: {
    email: TEST_USER_EMAIL,
    passwordHash,
    name: "Test User",
    role: "USER",
  },
});
```

Without this, every feature change has to re-invent the login fixture and tests break when someone changes the credentials.

## Order of Operations

1. **Delete in FK-safe order** — only if the seed intentionally wipes a subset (usually not needed after `db push --force-reset`).
2. **Create lookup tables first** — enums, categories, roles, permissions.
3. **Create users** — including the test user.
4. **Create aggregate roots** — products, orders, articles.
5. **Create dependent rows** — images, variants, line items.
6. **Never `await Promise.all([...createManyIndependent...])`** when later rows reference earlier IDs — race conditions on SQLite.

## Size Budget

Seed data is test fixture data, not production. Keep it small: handfuls of each entity, not hundreds. Large seed datasets slow every worktree's bootstrap (`prisma db push --force-reset && db seed` runs on every E2E bootstrap) and magnify the cost of any slow query in the seed script itself.
