# Capacitor Conventions

## Plugin Usage

- Install Capacitor plugins via npm/pnpm, then run `npx cap sync` to copy them to native projects.
- Always import from `@capacitor/<plugin>`, never from native code directly in web components.
- Check `Capacitor.isNativePlatform()` before calling native-only APIs — the app must also work in the browser.
- Prefer Capacitor's built-in plugins over community alternatives for stability.
- Dynamic imports for platform-specific plugins: `if (Capacitor.getPlatform() === 'android') { const { ShareTarget } = await import('@capgo/capacitor-share-target'); }`

## Platform-Specific Code

- Use `Capacitor.getPlatform()` to branch logic per platform (`'ios'`, `'android'`, `'web'`).
- Keep platform branches minimal — the web layer should be platform-agnostic.
- Never import `ios/` or `android/` files from web code.
- Dual-mode storage pattern: use Capacitor `Preferences` on native, `localStorage` on web. Always gate with `Capacitor.isNativePlatform()`.

## Native Configuration

- `capacitor.config.ts` is the single source of truth for Capacitor settings.
- `webDir` must match the Next.js build output directory (`out/` for static export). Mismatch causes blank screens on device.
- Plugin configuration goes in `capacitor.config.ts`, not in native code.
- Static export: use `output: 'export'` in `next.config.js` to generate `out/`. Server-side features (API routes, middleware) are NOT available in the native app — only in the web version or via a separate backend.

## Build Pipeline

- Build order: `npm run build` → `npx cap sync` → native build (Xcode/Gradle).
- `cap sync` copies web assets + syncs plugin native code — always run after web build.
- Never commit `ios/App/App/public/` or `android/app/src/main/assets/public/` — these are generated by `cap sync`.
- iOS archive: `xcodebuild archive` → `.xcarchive` → `.ipa` for distribution. Never commit `.xcarchive` or `DerivedData/`.

## iOS Specifics

- Info.plist permission strings must be user-facing and descriptive (App Store requirement).
- App Groups require matching identifiers in both main app and extensions. Check both `.entitlements` files.
- URL schemes for deep linking go in Info.plist, not in web code.
- Capacitor manages dependencies via SPM (Swift Package Manager), not CocoaPods. The `ios/App/Pods/` directory should not exist in modern setups.

## Share Extension

- iOS Share Extensions have a **25-second execution limit** — long operations must be deferred to the main app.
- Share Extensions run in a separate process — they cannot access the main app's state directly.
- Use App Groups (shared `UserDefaults` suite) to pass data between the Share Extension and the main app.
- The Share Extension should have its own API authentication (device ID from App Groups), not rely on web session cookies.
- Pattern: Extension saves shared data to App Groups → main app detects on `appStateChange` resume → processes the data.

## App Groups & Inter-Process Communication

- App Group identifiers must be consistent across all targets (main app, extensions). Use the format `group.<bundle-id>`.
- Both the main app and extensions need the App Group in their `.entitlements` file.
- AppDelegate should sync identity tokens (device ID, auth tokens) to the shared container on launch and on value changes.
- When injecting data from native into WebView, use base64 encoding to safely pass JSON through JavaScript evaluation.
- Never store sensitive secrets (API keys, passwords) in App Groups — use Keychain with shared access group instead.

## Deep Linking

- Register custom URL schemes in `Info.plist` under `CFBundleURLTypes`.
- Handle incoming URLs in AppDelegate or via Capacitor's `@capacitor/app` `appUrlOpen` listener.
- Use URL scheme callbacks to return control from extensions to the main app (e.g., `myapp://import-complete`).
- Web-side: listen for `App.addListener('appUrlOpen', ...)` and route accordingly.
