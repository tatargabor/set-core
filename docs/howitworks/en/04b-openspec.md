# OpenSpec — The Spec-Driven Workflow

## What Is OpenSpec?

OpenSpec is a structured development workflow that guides every change through phases before implementation begins. The goal: the AI agent should not "just code" but *understand* the task and make traceable decisions.

The phases:

```
Proposal (WHY)  →  Design (HOW)  →  Specs (WHAT)  →  Tasks (DO)  →  Implementation
```

## The Artifacts

### Proposal

The *why* document: motivation, scope, success criteria. The orchestrator **pre-generates** the proposal in the worktree from the plan's roadmap item and scope — the agent receives this as a starting point.

### Design

The *how* document: technical decisions in ADR (Architecture Decision Record) format, risks, alternatives. Generated by the agent from the proposal.

### Specs

The *what* document: detailed requirements in BDD (Given/When/Then) format, delta operations (ADDED, MODIFIED, REMOVED). Every requirement has testable scenarios.

### Tasks

The *do* list: a checkbox task list that the agent checks off one by one during implementation. The `[?]` marker indicates a manual (human) task — API key procurement, external service setup.

```markdown
- [x] Add JWT middleware to /api/* routes
- [x] Create login endpoint with bcrypt
- [ ] Add refresh token rotation
- [?] Configure OAuth provider credentials
```

## Integration with Orchestration

### Dispatch Phase

`dispatcher.sh` during worktree creation:

1. Creates the change directory (`openspec new change`)
2. **Pre-generates the proposal** from the plan scope
3. Copies spec context (in digest mode)
4. Appends cross-cutting file list, sibling change statuses, and memory context

### Ralph Loop Integration

The Ralph loop decides based on OpenSpec artifact state:

```
If no tasks.md → action: ff (fast-forward, create all artifacts)
If tasks.md exists → action: apply (implementation)
If everything done → action: done
```

In the `ff` phase, the agent runs the `/opsx:ff` skill: creates design, specs, and tasks from the proposal in a single session. In the `apply` phase, it runs `/opsx:apply`: implements tasks from tasks.md.

### FF→Apply Chaining

Previously, artifact generation (ff) and implementation (apply) ran in separate iterations, and the iteration boundary caused context loss. The current version chains them within a single iteration: after ff completes, apply starts immediately with fresh context.

## Interactive vs. Orchestrated Usage

| Mode | Commands | When |
|------|----------|------|
| **Interactive** | `/opsx:new` → `/opsx:continue` → `/opsx:apply` → `/opsx:verify` → `/opsx:archive` | Developer working alone, step by step |
| **Fast-forward** | `/opsx:ff` → `/opsx:apply` → `/opsx:archive` | Developer, but no time for step-by-step |
| **Orchestrated** | Automatic — dispatcher and Ralph loop handle it | `wt-orchestrate start` |

## Traceability

OpenSpec provides a complete traceability chain:

```
Spec section (v3.md)
  → Requirement ID (REQ-042)
    → Change assignment (platform-dashboard)
      → Proposal: "This change covers REQ-042"
        → Tasks: "Add KPI cards section"
          → Git commit: code implementing REQ-042
```

The verify gate checks not just code quality but **requirement compliance**: the review prompt includes the assigned REQ-XXX identifiers.

\begin{keypoint}
OpenSpec is the contract between orchestration and execution. The orchestrator creates the proposal; the agent creates the design, specs, and tasks; the verify gate checks the implementation. If any phase is missing, the system knows where it stands and what the next step is.
\end{keypoint}

## Limitations

- **FF failures**: `/opsx:ff` sometimes fails to create tasks.md (large spec, truncated LLM response). In such cases, the loop generates fallback tasks from the proposal
- **Manual tasks**: Tasks marked with `[?]` (API keys, external setup) require human intervention — orchestration cannot automate them
- **Large changes**: 14+ requirements in a single change → unreliable. The planner applies a max 6 REQ/change rule
