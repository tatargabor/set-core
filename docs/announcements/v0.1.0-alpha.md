## LinkedIn (English, professional, story arc)

**Release: set-core v0.1.0-alpha — spec-driven orchestration for Claude Code**

Over the last few months I've been building an orchestration framework for Claude Code that turns a markdown spec into merged features. Today I'm opening it as an early alpha so others can start breaking it.

**What it does.** You hand it a spec, it decomposes the spec into independent changes via OpenSpec, dispatches parallel Claude Code agents into isolated git worktrees, runs quality gates on each (build → test → e2e → review → rules → spec-verify), and merges the ones that pass. A web dashboard at localhost shows every agent's progress in real time. A supervisor process ("sentinel") watches the whole thing, investigates stuck changes autonomously, and restarts them — so you can walk away from a run.

**Why it exists.** Prompt-and-pray workflows don't scale to real codebases. I wanted a system that starts every change with a written requirement, verifies the output against that requirement, and only merges when the spec says the change is done. That's OpenSpec ("we don't prompt — we specify"), which set-core wires end-to-end into the Claude Code runtime.

**How far it is.** It's early alpha, with known rough edges (honest list at docs/alpha-release.md). But it's been dogfooded hard: set-core was built using itself — 376 capability specifications, 1,500+ commits, every one planned through OpenSpec and validated through the gates I'm describing. In the reference run ending today, it shipped a full coffee-webshop (auth, catalog, cart, checkout, admin, subscriptions, reviews) from a single markdown spec plus a Figma link, across 12 parallel changes.

**What works now.** Next.js project type (Prisma + NextAuth + Playwright + shadcn/ui), spec-driven planning, parallel agents in worktrees, integration gates with retry budgets, design-pipeline (Figma → design tokens → per-change visual briefs scope-matched to each change), web dashboard with activity timeline, MCP memory server, autonomous issue investigation.

**What's coming.** A design-compliance gate that LLM-reviews rendered Playwright screenshots against design tokens. Postgres per-worker E2E isolation. More project type plugins.

The point of going out now is that others start using it. There will be bugs. The sentinel takes care of most of them in practice.

Try it: https://github.com/tatargabor/set-core
Website: https://setcode.dev

If you build with AI agents and you're tired of prompting, I'd love your feedback.

## Anthropic Discord / community (shorter, technical, for people who already know Claude Code)

**set-core v0.1.0-alpha — spec-driven multi-agent orchestration on top of Claude Code**

I built an orchestration framework for Claude Code and shipped it as early alpha today. Going out now because the point is for other people to try it.

**The loop:** spec (markdown) → OpenSpec digest → parallel Claude Code agents in isolated git worktrees → quality gates per change (build, test, e2e, LLM code-review, rules check, spec-verify) → merge. A sentinel supervises, auto-investigates stuck changes, and re-dispatches. A web dashboard at localhost:7400 shows real-time gate progress, activity timeline, tokens, per-change drilldown.

**Built with itself.** 376 capability specs, 1,500+ commits, all driven through the same pipeline. The reference end-to-end run completing today shipped a full Next.js webshop (12 changes: auth, catalog, cart, checkout, admin, subscriptions, reviews, ...) from one markdown spec plus a Figma link.

**What's there:**
- Web project type (Next.js 14 + Prisma + NextAuth v5 + Playwright + shadcn/ui)
- Session resume across dispatcher retries (keeps Claude cache warm)
- Integration gates with retry budgets and auto-replan
- Design pipeline: Figma → design tokens → scope-matched per-change visual briefs
- MCP memory server + autonomous issue investigation

**What's alpha about it:**
- ~15% change failure rate on larger plans (2 of 12 in the reference run)
- One dispatch-state race that the sentinel covers but the engine hasn't root-caused yet
- Opus-by-default in the shipped test configs (override to sonnet for cost)

## OpenSpec community (emphasize the OpenSpec integration)

**set-core v0.1.0-alpha — Claude Code orchestrator built on OpenSpec**

Early alpha of set-core is public: a framework that uses OpenSpec end-to-end as the contract between a human spec and automated AI implementation.

**How OpenSpec fits in.** Every change in set-core starts as an OpenSpec proposal, gets a design.md, specs with requirements + acceptance criteria, and a tasks.md — all before any code is written. The orchestrator then dispatches a Claude Code agent into a git worktree with the change's artifacts already in place, and the agent implements against them. A spec-verify gate checks the implementation against the requirements before merge. If it fails, the finding goes back into the agent's retry context.

**What set-core adds on top of OpenSpec:**
- Plan decomposition: a single spec becomes N OpenSpec changes with dependency ordering
- Parallel dispatch of independent changes into git worktrees
- Quality gates: build, test, e2e, LLM code-review, rules, spec-verify
- Sentinel supervisor that auto-recovers from stuck changes
- Web dashboard with real-time gate progress
- 352 archived changes so far in its own development — all through OpenSpec

**Why I'm telling you.** set-core is probably the heaviest OpenSpec consumer that exists right now (376 specs in its own repo), and we've hit a few edges in practice — integration patterns, 1.1 → 1.2 migration, scaling rule coverage through multi-feature plans. Happy to contribute learnings back.
