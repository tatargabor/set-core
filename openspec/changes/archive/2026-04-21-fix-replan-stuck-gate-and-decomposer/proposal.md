## Why

The `craftbrew-run-20260418-1719` E2E run exposed nine framework-level defects that together burned ~3 hours of wall time with zero new commits. After a replan created a brand-new 6-change decomposition, stale state from the previous plan remained entangled (old worktrees, branches, `openspec/changes/` dirs), so the orchestrator could never align with the new plan. The first real quality-gate run (on `promotions-engine`) exposed that the engine and verifier have two handlers that contradict each other when an agent exits its fix loop with `loop_status=stuck`: the engine says "route to re-gate" while the verifier marks the change `stalled` for the watchdog. The watchdog then auto-recovers every 300s, the agent runs briefly, exits stuck, and the loop repeats indefinitely with no termination and no progress detection (observed 23 times, ~71M input tokens consumed on one change). On top of this, the supervisor fired `log_silence` triggers every 15s during startup even after `retry_budget_exhausted` (pure noise), and the gate registry selected gates by static `change_type` rather than by the change's actual content — `foundation-setup` contained UI copied from `v0-export` but design-fidelity and e2e gates never ran on it. Finally, decomposed changes are still too coarse: one change routinely spans 15+ capabilities, which hides UI content behind infrastructure labels and makes retry-budget math meaningless.

## What Changes

- **Replan state reconciliation** — When a replan produces a change-name set that diverges from the prior plan, archive-then-force-remove stale worktrees (even dirty ones, via `git stash -u` + archive path), delete orphan `change/*` branches, and remove `openspec/changes/<name>/` dirs that are neither in the new plan nor in `state-archive.jsonl`. Reconciliation SHALL run **after** `collect_replan_context()` has captured its snapshot of completed/merged changes, so archived-as-stale dirs do not pollute the replan prompt's `completed_names`. Reset the supervisor's `rapid_crashes` counter on clean plan completion.
- **Stuck re-gate handler — single path** — When the agent exits its fix loop with `loop_status=stuck` + new commits, the engine SHALL re-run the verify gate directly on the next monitor poll and SHALL NOT mark the change `stalled`. The verifier's stale-detection branch (the code path that writes `status=stalled` when `ralph_status ∈ {stopped, stalled, stuck}`) SHALL additionally check for new commits since `stalled_at`; if new commits exist, it SHALL skip the stall write and let the engine's re-gate path run. Introduce a per-change `stuck_loop_count`; after N (default 3) consecutive stuck exits without a gate-state change (same `stop_gate` + same finding fingerprint), fail the change hard and surface it to the investigation/fix-iss pipeline.
- **Retry-budget escalation to investigation** — When the verify gate's retry budget for a given `stop_gate` is exhausted (e.g. review 5/5), automatically create a `fix-iss-<n>-<slug>` change via the same machinery that produced `fix-iss-004` in the prior run, and switch dispatch to it. No more infinite retries on the same change.
- **Supervisor exponential back-off on retry-budget-exhausted** — When a `SUPERVISOR_TRIGGER` is `skipped: retry_budget_exhausted`, back off the trigger exponentially (60s → 120s → 240s → … → cap 600s) per (trigger, change, reason) tuple instead of re-firing every 15s.
- **Per-change token circuit breaker** — If a single change's cumulative `input_tokens` grows by more than a configurable delta (default 20M) without a change in the most recent gate state (`stop_gate`+`finding_fingerprint`), pause the change, mark it `failed:token_runaway`, and surface to investigation.
- **Content-aware gate selection** — Replace the static `change_type → gate_set` mapping with a content-scanning selector that reads the dispatch file scope + the first commit's diff to decide which gates are relevant. If the scope or diff touches UI patterns (`src/app/**/*.tsx`, `components/**`, `*.css`), design-fidelity and e2e-ui SHALL be added to the gate set regardless of the change's label. If the scope is server-only logic, unit tests SHALL be required. Re-detection runs once after the agent's first commit and can add (never remove) gates.
- **Decomposer skip_test guard** — The decomposer SHALL NOT set `skip_test=true` on changes whose scope includes server-side business logic (validators, actions, handlers, services). Skip_test is allowed only for pure scaffolding, docs, or config-only changes.
- **Decomposer size-estimate formula + linked sibling split** — Replace hard caps (req/scenario/file counts correlate poorly with effort) with a profile-weighted `estimated_loc` score: `Σ(loc_weight(path) for path in scope.files) + models*120 + ambiguities*80`. Default threshold 1500 LOC; over-threshold changes auto-split into **linked sibling changes** sharing the base name (`<base>-<group>-<N>`), chained via `depends_on` in the same phase. Web profile declares path weights (e.g., `src/app/admin/**/page.tsx`→350, `src/app/[locale]/**/page.tsx`→200). Calibrated against the reference 6-change plan: admin-operations (est 3760) → 3 siblings (orders/dashboard/returns); promotions-engine (2000) → 2 siblings (server/admin); reviews-and-wishlist (1640) → 2 siblings (customer/admin); email-and-seo (1750) → 2 siblings (delivery/polish). Subscription-flow and stories-and-content pass through unchanged. Result: 6 → 11 changes, each ~1000 LOC, fitting one opus session without retry-budget strain.
- **Web module gate tuning** — `i18n_check` SHALL be a hard-fail gate (not warn-fail). Parallelise `spec_verify` and `review` since they don't depend on each other; target p50 gate-run wall time ≤120s.
- **Per-gate retry policy (scoped re-gate)** — Retries are the second-largest time sink after coarse changes. Add a profile-declared `gate_retry_policy()` with three modes: `always` (re-run fully — default, safe), `cached` (reuse prior verdict unless invalidated by diff-touches-scope, cache-use cap, or new-API-surface detection), `scoped` (run on a subset filtered by retry diff — e.g., Playwright only runs tests whose pages transitively import the touched files). Web profile declares: cheap gates (build/lint/test/i18n/scope_check) → `always`; expensive LLM gates (review/spec_verify/design-fidelity) → `cached`; e2e → `scoped`. Cache forced to invalidate after 2 consecutive reuses (safety valve). Retry prompt gains a "Cached Gates" section so the agent knows which prior verdicts still hold. Also adds a cumulative **retry wall-time budget** (default 30 min/change) that escalates to fix-iss on exhaustion — guards the "slow death by many small retries" failure mode.

## Capabilities

### Modified Capabilities
- `orch-replan-python` — divergent-plan hard-reset cleanup on replan
- `orphan-cleanup` — force-clean dirty worktrees when they don't appear in the new plan
- `retry-loop-completion` — single stuck-exit handler + stuck_loop_count circuit breaker
- `supervisor-transition-triggers` — exponential back-off on retry_budget_exhausted
- `decomposition-agent` — skip_test guard + granularity budget + content hints for gate selection
- `per-change-gate-skip` — content-scan-driven gate inclusion (not just exclusion)
- `web-gates` — i18n hard-fail + spec_verify/review parallelisation
- `orchestration-token-tracking` — per-change input-token runaway circuit breaker
- `investigation-runner` — auto-create fix-iss on retry-budget exhaustion

### New Capabilities
- `gate-retry-policy` — profile-declared per-gate retry behavior (always / cached / scoped) with safety valves and cache-invalidation rules.

## Impact

- **Core engine**: `lib/set_orch/engine.py` (stuck handler, replan cleanup hook), `lib/set_orch/verifier.py` (remove `marking stalled` branch on stuck), `lib/set_orch/dispatcher.py` (stuck_loop_count), `lib/set_orch/orphan_cleanup.py` (force-clean divergent-plan worktrees), `lib/set_orch/gate_registry.py` (content-scan selector), `lib/set_orch/supervisor/anomaly.py` (back-off), `lib/set_orch/planner.py` / decomposition prompts (granularity budget, skip_test guard).
- **Web module**: `modules/web/profile.py` (gate tuning), `modules/web/gates/i18n_check.py` (hard-fail).
- **State schema**: `orchestration-state.json` gains `stuck_loop_count`, `last_gate_fingerprint`, `token_runaway_baseline`, `fix_iss_child`, `gate_recheck_done`, `touched_file_globs`, `retry_wall_time_ms`, `verify_retry_index`, `gate_retry_tracking` per change; `supervisor-status.json` gains `trigger_backoffs` per `(trigger, change, reason)` tuple. All backwards-compatible (default None/0/{} on old state).
- **Dispatcher contract**: `gate_set` is now a function of `(change, touched_files, first_commit_diff)`, not `change_type` alone. Re-dispatch after first commit may enlarge the gate set; this is logged.
- **No changes** to MCP surface, web API, or CLI flags. Internal invariant only.
