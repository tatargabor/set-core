## Context

The orchestration pipeline follows a spec→tasks→apply→verify flow. Agents implement changes based on proposal.md scope and tasks.md checklists, then the verify gate runs build/test/review/spec-verify checks before merging.

Current weaknesses identified through production runs:
1. Specs have no explicit scope boundary — agents decide what to implement based on interpretation
2. Tasks don't reference requirement IDs — traceability breaks at the task level
3. Verify gate only checks forward (spec→code) via keyword search, never backward (code→spec)
4. Soft-pass logic auto-passes spec verification when the sentinel string is missing
5. WHEN/THEN scenarios exist in specs but are never translated into testable acceptance criteria
6. `validate_plan()` checks that referenced REQ IDs exist but NOT that all digest REQs are covered — requirements silently fall through the plan

## Goals / Non-Goals

**Goals:**
- Reduce hallucinated features (overshoot) through scope boundaries and backward verification
- Reduce missed requirements (undershoot) through traceability and acceptance skeletons
- Fix the soft-pass bypass in verifier.py
- All changes backward-compatible — existing changes without new sections degrade gracefully

**Non-Goals:**
- Machine-parseable spec format (keep human-readable WHEN/THEN markdown)
- Automated test generation (generate skeletons only, agent fills in assertions)
- No new gate steps in verifier.py and no reordering of the existing gate sequence (modifying result semantics of existing steps IS in scope — e.g., the soft-pass fix)
- Per-task file scope restriction (tracking which files each task is allowed to touch — distinct from the feature-level IN SCOPE / OUT OF SCOPE boundary added to the spec template)

## Decisions

### Decision 1: Scope boundary in spec template, not proposal

Add IN SCOPE / OUT OF SCOPE to the **spec** artifact template (not proposal). Rationale: specs define requirements at the capability level — scope boundaries belong with requirements, not with the high-level "why" document. The verify skill checks code against spec scope boundaries.

Alternative considered: Adding to proposal.md — rejected because proposals are intentionally brief and don't define testable requirements.

### Decision 2: Lightweight traceability — tags in task text, not structured JSON

Use inline `[REQ: requirement-name]` tags in task descriptions rather than a separate JSON mapping file. Rationale: tasks.md is human-editable markdown consumed by agents — inline tags are visible in context without needing to cross-reference another file. The ff-change skill instruction tells the agent to add tags; the verify skill parses them for the traceability matrix.

Format: `- [ ] 1.1 Create login form [REQ: user-authentication]`

Alternative considered: Separate `traceability.json` — rejected because it adds a file that can drift from tasks.md and increases cognitive load.

### Decision 3: Traceability matrix as verify output, not tasks.md section

The traceability matrix (requirement → tasks mapping) is generated dynamically by the verify skill during verification, not statically in tasks.md. Rationale: if the matrix is in tasks.md, it can be out of sync with actual task content. Generating it during verify ensures it reflects reality.

The verify skill outputs:
```
## Traceability Matrix
| Requirement | Tasks | Status |
|-------------|-------|--------|
| REQ-NAME    | 1.1, 1.3 | Covered |
| REQ-OTHER   | (none) | MISSING |
```

### Decision 4: Overshoot detection via diff analysis in verify skill

The verify skill (not verifier.py) performs overshoot detection by:
1. Reading the spec's IN SCOPE section to get the allowed scope
2. Scanning the diff for new exports, routes, endpoints, components, database tables
3. Checking each against the spec requirements
4. Flagging items not traceable to any requirement as WARNING

This runs inside the `/opsx:verify` Claude session (Step 6 of the gate pipeline), not as a separate gate step. Rationale: overshoot detection requires LLM judgment (is this new route in spec scope?) — pure regex/glob matching would produce too many false positives.

### Decision 5: Fix soft-pass by treating missing sentinel as explicit FAIL

Replace the soft-pass logic (lines 1623-1642 of verifier.py) with:
- Missing VERIFY_RESULT sentinel → `spec_coverage_result = "timeout"`
- Log a warning but treat as FAIL for blocking gates
- Non-blocking gates still continue

This removes the dangerous path where a timed-out spec verify auto-passes because build/test happened to pass.

### Decision 6: Acceptance skeleton as a section in tasks.md, not separate file

Add an `## Acceptance Criteria` section at the bottom of tasks.md, generated by the ff-change skill from WHEN/THEN scenarios. Format:

```markdown
## Acceptance Criteria (from spec scenarios)

- [ ] AC-1: WHEN valid credentials THEN redirect to / [REQ: user-auth, scenario: successful-login]
- [ ] AC-2: WHEN invalid password THEN show error [REQ: user-auth, scenario: invalid-credentials]
```

The apply agent checks these off as it implements. The verify skill treats unchecked AC items as CRITICAL (same as unchecked tasks).

Alternative considered: Separate `acceptance-tests.md` — rejected to avoid file proliferation and keep everything the apply agent needs in one place.

### Decision 7: Plan completeness check in validate_plan() with deferred_requirements

Add reverse requirement coverage check to `validate_plan()`: compute the set difference between all digest REQ IDs and all REQ IDs assigned across changes (requirements[] + also_affects_reqs[]). Unassigned REQs are checked against a `deferred_requirements` array in the plan JSON.

Plan schema addition:
```json
{
  "changes": [...],
  "deferred_requirements": [
    {
      "id": "REQ-EXPORT-001",
      "reason": "Depends on auth system, planned for next phase"
    }
  ]
}
```

Validation logic (reads `all_digest_reqs` from `digest_dir/requirements.json`):
- `unassigned = all_digest_reqs - (all change requirements[] ∪ all change also_affects_reqs[])`
- `unexplained = unassigned - deferred_requirement_ids`
- If `unexplained` is not empty → `result.errors.append()` (ERROR, not warning — blocks dispatch)
- If `deferred` is not empty → `result.warnings.append()` (informational — logged but doesn't block)

This runs in `validate_plan()` which is called BEFORE dispatch. A re-plan is ~30 seconds vs a verify retry at ~1-3 hours.

Alternative considered: Adding a separate "plan review" gate step — rejected because `validate_plan()` already runs before dispatch and is the natural place for this check.

### Decision 8: Decompose prompt instructs explicit deferral

The decompose skill prompt must instruct the planner to:
1. Account for every requirement from the digest
2. Either assign it to a change's `requirements[]` or `also_affects_reqs[]`
3. Or explicitly list it in `deferred_requirements[]` with a reason
4. Never silently drop a requirement

This is a prompt-level change in the decompose skill, not a code change.

## Risks / Trade-offs

- [Risk] Agents may ignore `[REQ: ...]` tags or generate them incorrectly → Mitigation: verify skill detects tasks without REQ tags and flags as WARNING; ff-change skill instruction makes tagging mandatory
- [Risk] IN SCOPE / OUT OF SCOPE adds verbosity to specs → Mitigation: keep it concise (bullet lists, not paragraphs); optional section that degrades gracefully
- [Risk] Overshoot detection false positives (flagging legitimate helper functions) → Mitigation: WARNING severity not CRITICAL; verify skill uses LLM judgment
- [Risk] Removing soft-pass may increase verify failures for valid changes where `/opsx:verify` times out → Mitigation: increase max-turns from 15 to 20 for spec verify; add retry budget specifically for spec verify timeout
- [Trade-off] Acceptance criteria in tasks.md increases file size → Acceptable: tasks.md is already the agent's primary working document
- [Risk] AC item inflation — large specs with many scenarios could generate 30+ AC items, increasing tasks.md size and agent context → Mitigation: the ff-change skill should note that AC items are generated from scenarios in the change's own delta specs only, not from all specs in the project
- [Risk] `[REQ: ...]` tag parsing fragility — variations in whitespace or casing (`[REQ:name]` vs `[REQ: name]`) could break traceability → Mitigation: verify skill parser must be whitespace-tolerant and case-insensitive; canonical format defined as `[REQ: kebab-case-slug]`
- [Risk] Plan schema backward-compatibility — `deferred_requirements[]` is a new field in plan JSON → Mitigation: field is optional with default `[]`; older validators skip unknown fields; plan JSON is not persisted across wt-tools versions
- [Risk] Plan completeness check may be overly strict for partial/incremental planning → Mitigation: only enforce when `digest_dir` is provided; `deferred_requirements` provides the explicit escape hatch
- [Risk] Decompose agent may generate low-quality deferral reasons → Mitigation: reasons are informational (warnings), not blocking; the value is forcing explicit acknowledgment rather than silent omission
